The Build Process

Why it exists and how it works

Why This Exists

The first time Wallet was built, the code was written before the test plan existed. Fixtures were never created — the frontend invented its own mock shapes instead of deriving them from the API Contract. The result: every monetary value displayed 100× too high, the trust badge was always wrong, withdrawal history never appeared. These were not edge cases. They were visible on every screen, for every user. They were caught in QA — but only because the team ran a full test. Without the test, they would have reached real CSPs.

The process below exists to make these bugs structurally impossible, not just testable.

The 7 Steps
STEP 1
DEFINE
Fill Module Design Template v1.2. Lock the user reality, system truth, and state model before any build work begins.
Skipping causes: building features instead of state interfaces. Wrong module built entirely.
STEP 2
DESIGN — Two Parallel Tracks
Track B (Backend): Build Spec → Devender builds service → API Contract finalized.
Track A (Frontend): UX Agent → Visual Spec → Copy Agent → Hindi keys → Product creates fixtures + render checklist → Frontend builds.

The two tracks run in parallel but frontend BUILD cannot start until the API Contract exists.
Skipping Track A fixtures causes: frontend invents mock shapes. Real data doesn't match. 100× paise bug.
STEP 3
TEST PLAN
Fill Module Test Strategy Template v1.0. Define L1 truth cases, L2 architecture cases, L3 integration cases, L4 user scenarios. Canonical fixtures created here.
Skipping causes: tests not reproducible. Different results every run.
STEP 4
EXECUTE — Sequential Gates
E1: Developer tests before merge.
E2: Integration tests after merge (100% pass required).
E3: Manual QA on real budget Android device, Hindi locale, 3G.
E4: Field test — 3-5 real CSPs, task in Hindi, no training, just watch.

Each blocks the next. Cannot skip to E3 without E2 passing.
Skipping causes: bugs found by real CSPs that would have been caught in controlled QA.
STEP 5
GATES — All 5 Must Pass
G1 Truth: all states correct, all actions valid, forbidden behaviors absent.
G2 Architecture: no local logic, no derived actions, 15 L2 rules pass.
G3 Integration: contract fields complete, idempotency works, cross-module effects visible.
G4 User: CSP answers 2-second test, completes task, no surprise.
G5 Performance: first paint <2s on 3G, no blocking navigation, 60fps scroll.
Skipping any gate causes: structural defects shipped to production.
STEP 6
SHIP — Staged
Stage 1: Internal staff (5-10), 3-5 days, metrics clean.
Stage 2: Controlled cohort (20-50 CSPs), 1-2 weeks, metrics clean.
Stage 3: All CSPs, gradual.
Skipping stages causes: undetected issues reaching full CSP base simultaneously.
STEP 7
REGRESSION
After every module ships: create 10-15 critical scenarios. These run before every future module release. Module 4 cannot ship if it breaks Module 3's regression pack.
Skipping causes: future builds silently break shipped modules.

The Amendment Process

When something breaks or changes, this determines how it's fixed.

Q1: Can you point to an exact spec section that defines the correct behavior?
YES CODE_BUG (Type 1) Devender fixes code. Spec is already correct.
NO Go to Q2.
Q2: Does the OS define the behavior even if the spec missed it?
YES SPEC_GAP (Type 2) Mukul fixes spec first. Then Devender fixes code.
NO Go to Q3.
Q3: Is this behavior defined nowhere?
YES DESIGN_GAP (Type 3) STOP. Satyam decides. Full amendment chain runs.
Rule: The documents do the classifying, not the people. If the engineer cannot point to a spec clause, they cannot call it a code bug.
Non-Negotiable Rules
  1. Builder never verifies their own work. The person who builds is never the person who verifies.
  2. Test plan must exist BEFORE Devender writes code. Phase A before Phase B.
  3. Canonical fixtures must exist BEFORE frontend builds. Product Phase A before Engineer Phase B.
  4. All 5 gates must pass. No partial shipping.
  5. Spec is truth, not the test. If test expects something spec doesn't say, test is wrong.
  6. Frontend never computes business logic. Server decides. App renders.
  7. Contract-driven fixtures only. No free-form mocks.
  8. Regression pack runs before every future module release.
  9. Amendment process governs all changes. Upstream always wins a contradiction.
  10. Builder never self-reviews. Adit cross-reviews all code and test plans (except PS test plan which Adit wrote — Mukul must review that one).
