STEP 3 TEST STRATEGY
Note: L1+L2 written during Track B Design (from Build Spec). L3+L4 + fixtures written after Contract locked. Must be COMPLETE before E1 can start.

L1 — Truth cases (15–30): Does service return correct values per spec?
L2 — Architecture cases (8–12): Does it follow architecture rules?
L3 — Integration cases (15–25): Frontend + backend together, contract honoured end-to-end.
L4 — User scenarios (5–10): Real user flows on the full module.
Canonical fixtures + seed personas: Derived from API Contract.

Owner: Mukul writes (L1+L2), Mukul+Satyam write (L3+L4), Product creates fixtures

STEP 5 SHIP
Staged rollout: Wiom staff (3–5 days) → 20–50 CSPs (1–2 weeks) → All CSPs
Block between stages if: crash >3%, success <85%, 3+ CSPs same issue, any balance mismatch
Gate: Only if ALL gates G1–G5 passed
STEP 6 REGRESSION
Create regression pack: 10–15 critical scenarios from L1+L3.
Add to central regression suite. Every future module PR runs regression for ALL shipped modules.
Any regression failure → BLOCK release.
What happens if you skip
If you skip…What breaks
Step 1 (Define)No scope boundary → scope creep during build. No success criteria → ship without knowing if it worked.
Track B Phase A (Build Spec before code)No defined expected behavior → code written freely → bugs only found in E1 or later.
CLAUDE.md before code genClaude invents behavior — no spec context, no architecture rules, no event list. Output is useless.
Test Strategy L1+L2 before codeNo truth reference before E1 → developer self-check is improvised → bugs reach integration.
Test Strategy L3+L4 + fixtures before E1E2 has no L3 cases to run. E3 has no L4 scenarios or canonical fixtures. QA becomes improvised.
G1 gate (E2 must follow E1 pass)Integration test runs against broken logic → cascading failures → expensive rework after merge.
E4 field testReal CSP usability issues only discovered in production — too late to fix cleanly.
E5 production monitoringSilent failures at scale — balance mismatches, crashes undetected until CSP reports them.
Regression Pack at shipFuture module releases break existing behavior — no safety net, each ship is a gamble.
Amendment Decision Tree
Something is wrong or missing → follow this tree before touching any code or spec
Q1: Does the spec describe the correct behavior?
YES → Code is wrong CODE_BUG Fix code, no spec change needed
NO → Q2: Does the OS/Architecture cover this scenario?
YES → Spec missed it SPEC_GAP Fix spec, code follows
NO → Q3: Is this a value/parameter only?
YES → PARAMETER_CHANGE Update config only
NO → Behavior undefined everywhere DESIGN_GAP STOP — escalate to Satyam
HOTFIX Active production harm → emergency track. Bypass normal flow only with Satyam approval.
Frontend Sub-types
RENDER_BUG
Wrong color, wrong badge rendering — fix in code, no spec change
COPY_BUG
Wrong Hindi text displayed — fix copy, check against contract
ARCHITECTURE_DRIFT
Computing locally what must be computed by backend. Always PR rejection — not "fix next sprint"
PERFORMANCE_REGRESSION
G5 gate failure — must be resolved before ship
VISUAL_SPEC_GAP
Spec is missing a rendering rule → STOP. Do not invent rendering. Escalate to Satyam.
CONTRACT_MISMATCH
Gateway returning wrong shape → backend fixes. Frontend does NOT workaround.
